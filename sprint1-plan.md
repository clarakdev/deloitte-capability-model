## Context

- Prototype: `project-management-prototype/` (Streamlit, SFIA 9, 30 synthetic employees)
- Sprint 1 builds the real backend in the root folder (alongside `project-management-prototype/`)

## User Stories

| ID | Story |
|----|-------|
| US001 | As a PM, I want to see a list of capabilities associated with each role |
| US002 | As a PM, I want the product to infer the top matching capabilities for each role automatically |
| US003 | As a PM, I want to add, remove and modify the capability list per role |
| US004 | As a PM, I want to weight certain capabilities more highly |
| US005 | As a PM, I want to restrict matching to employees with prior experience in that role |
| US006 | As a PM, I want to see available employees that best fit each role |
| US007 | As a PM, I want to see how well each candidate fits and why |

## Data Design Decisions

- **Skill framework**: ESCO v1.2.1. Curated subset filtered from `ESCO/skills_en.csv` by `reuseLevel` = `cross-sector` + `transversal`, with keyword exclusion for irrelevant domains (agriculture, performing arts, etc.). Initial quick filter done manually; proper subset to be refined with the team later.
- **ESCO files to keep**: `skills_en.csv`, `skillsHierarchy_en.csv`, `broaderRelationsSkillPillar_en.csv`, `occupations_en.csv`, `occupationSkillRelations_en.csv`
- **Employee data**: Full DPN profile format (all fields used for richer matching). Clients require employees keep profiles up-to-date, so sparse profiles are not expected. Empty fields are skipped at embedding time rather than embedded as empty strings (avoids noise).
- **DPN proficiency scale**: Binary (has skill / doesn't have skill) for this sprint. **⚠️ Confirm 1–5 scale vs binary with client before next sprint.**
- **Matching approach**: Direct embedding comparison — no explicit DPN→ESCO mapping step. The shared semantic space of the sentence-transformer handles cross-vocabulary similarity naturally.
  - **Role vector**: weighted sum of ESCO capability embeddings, where each capability's weight is the PM-assigned importance (1–5)
  - **Employee vector**: 0.6 × primary + 0.4 × secondary
    - Primary: mean of DPN skill name embeddings (binary presence)
    - Secondary: mean of [summary embedding, tools-list embedding, prior-roles embedding]
- **Gap analysis**: For each required ESCO capability, find the employee's closest DPN skill by cosine similarity. Flag as a gap if similarity < 0.6.
- **Availability**: Boolean `available` field for this sprint (extendable to capacity % or date-based later).
- **US005 filter**: Case-insensitive exact title match against employee's `prior_roles` list.
- **US003 modify**: Only allows switching to or adding skills from the ESCO list — no freetext custom capabilities.
- **Demo project**: Government Digital Services Modernisation (5 roles from prototype usage guide).
- **Synthetic employees**: ~30 employees, all DPN fields populated. Generated by script — Gemini API for professional summaries, templates for structured fields.
- **ESCO attribution**: Frontend team must include the following statement in the UI: *"This service uses the ESCO classification of the European Commission."* Plus a note that the dataset has been filtered. Required by Commission Decision 2011/833/EU.

## Architecture

### File Structure
```
/ (root)
  app.py                          - FastAPI application + in-memory capability state
  requirements.txt
  core/
    __init__.py
    embedding_engine.py           - sentence-transformer singleton + ESCO cache loader
    capability_inference.py       - role description → top-N ESCO capabilities
    employee_profile.py           - load DPN employee, build composite embedding
    matching.py                   - rank employees per role (with weights + filters)
    gap_analysis.py               - per-capability coverage/gap explanation
  data/
    esco_skills.csv               - curated ESCO subset (filtered from ESCO/)
    esco_embeddings.npy           - pre-computed embeddings (from script)
    esco_codes.json               - cache validation list
    employees.json                - ~30 synthetic DPN-format employees
    project.json                  - pre-defined demo project + roles
  scripts/
    precompute_embeddings.py      - builds ESCO embedding cache
    generate_employees.py         - generates synthetic employee data
  tests/
    test_capability_inference.py
    test_matching.py
    test_gap_analysis.py
```

### API Endpoints

| Method | Path | Description | US |
|--------|------|-------------|----|
| GET | `/project` | Full project + roles | US001 |
| GET | `/roles/{id}/capabilities` | Auto-inferred on first call; returns editable list thereafter | US001, US002 |
| POST | `/roles/{id}/capabilities` | Add a capability by ESCO URI | US003 |
| PUT | `/roles/{id}/capabilities/{cap_id}` | Update weight (1–5) or swap ESCO skill | US003, US004 |
| DELETE | `/roles/{id}/capabilities/{cap_id}` | Remove a capability | US003 |
| GET | `/esco/search?q=...` | Search ESCO skills by name | US003 |
| GET | `/roles/{id}/candidates` | Ranked candidate list (`?require_prior_experience=bool&available_only=bool`) | US005, US006 |
| GET | `/roles/{id}/candidates/{emp_id}/fit` | Per-capability fit breakdown | US007 |

### In-Memory State
A dict keyed by `role_id` storing each role's current capability list. Initialised lazily on the first `GET /roles/{id}/capabilities` call by running capability inference. Mutations via POST/PUT/DELETE update this dict.

## Key Data Schemas

### Employee (DPN format) — `data/employees.json`
```json
{
  "id": "EMP001",
  "name": "Jane Smith",
  "title": "Senior Consultant",
  "role_level": "Senior",
  "business_unit": "Technology",
  "location": "London",
  "summary": "Narrative of experience and strengths...",
  "years_experience": 7,
  "current_role": "Senior Consultant",
  "prior_roles": ["Project Manager", "Business Analyst"],
  "project_experience": ["Cloud migration", "ERP implementation"],
  "industry_experience": ["Banking", "Technology Services"],
  "skills": [
    {"name": "Project Management", "category": "Business Skills"},
    {"name": "JIRA", "category": "Technology Skills"}
  ],
  "tools": ["JIRA", "Planview", "SAP"],
  "certifications": ["Prince2", "CAPM"],
  "available": true
}
```
> Note: `skills` is binary (present = employee has it). No proficiency level this sprint.

### Role Capability (in-memory, not persisted)
```json
{
  "cap_id": "http://data.europa.eu/esco/skill/...",
  "name": "project planning",
  "esco_description": "...",
  "embedding": [...],
  "weight": 3,
  "is_inferred": true
}
```

### `GET /roles/{id}/candidates` response item
```json
{
  "employee_id": "EMP001",
  "name": "Jane Smith",
  "title": "Senior Consultant",
  "match_score": 0.82,
  "available": true,
  "has_prior_experience": true
}
```

### `GET /roles/{id}/candidates/{emp_id}/fit` response
```json
{
  "employee_id": "EMP001",
  "overall_score": 0.82,
  "capability_coverage": [
    {
      "required_capability": "project planning",
      "weight": 4,
      "closest_employee_skill": "Project Management",
      "similarity": 0.88,
      "is_gap": false
    }
  ]
}
```

## Implementation Phases

### Phase 1 — Data & Infrastructure *(can be started in parallel)*
1. Filter `ESCO/skills_en.csv` → `data/esco_skills.csv` (cross-sector + transversal reuseLevel, domain keyword exclusions)
2. Write `scripts/precompute_embeddings.py` → `data/esco_embeddings.npy` + `data/esco_codes.json`
3. Write `scripts/generate_employees.py` → `data/employees.json` (~30 employees, all fields)
4. Create `data/project.json` (Government Digital Services Modernisation + 5 roles)

### Phase 2 — Core Engine *(steps 5–9, mostly parallel after step 5)*
5. `core/embedding_engine.py` — sentence-transformer singleton + ESCO cache loader *(adapt from `project-management-prototype/core/embedding_engine.py`)*
6. `core/capability_inference.py` — embed role description → cosine sim against ESCO embeddings → top-N capabilities *(adapt from `project-management-prototype/core/role_mapper.py`)*
7. `core/employee_profile.py` — build composite employee embedding from DPN fields (skip empty fields)
8. `core/matching.py` — weighted cosine similarity, availability filter, US005 prior-experience filter, sort *(adapt from `project-management-prototype/core/matching.py` `rank_employees()`)*
9. `core/gap_analysis.py` — per-capability coverage, gap threshold 0.6 *(adapt from `project-management-prototype/core/gap_analysis.py`)*

### Phase 3 — API Layer *(depends on Phase 2)*
10. `app.py` — FastAPI app, Pydantic models, in-memory state, all endpoints

### Phase 4 — Tests
11. `tests/test_capability_inference.py`
12. `tests/test_matching.py`
13. `tests/test_gap_analysis.py`
14. Update `requirements.txt` (add `fastapi`, `uvicorn`, `pydantic`)
